# Viewer 主图与胶卷带问题复盘

这份文档记录 2026-03-20 这轮 viewer 交互修复后的当前状态、根因和后续约束，避免同类问题再次回到仓库里。

## 问题现象

### 1. 主图组件尺寸失控

- 在导入大量 frame 后，group viewer 的主对比图会严重超出屏幕。
- 浏览器 DevTools 中能看到对比图实际渲染尺寸被拉大到类似 `14939.982 × 8402.922`。
- 这个尺寸比例仍然是 `16:9`，说明不是图片纵横比错了，而是外层布局宽度被错误放大了。

### 2. 胶卷带“能拖但不是真滚动”

- frame 数量少时问题不明显。
- frame 数量一多，底部胶卷带虽然看起来可以拖拽，但鼠标拖动不能稳定滚到后面的缩略图。
- 行为上更像是一个半失控的拖拽轨道，而不是可靠的横向滚动容器。

## 根因

### 1. 主图宽度被胶卷带轨道的 max-content 宽度带大

旧实现里，胶卷带使用第三方轨道拖拽方案，缩略图行本身有很强的内容宽度。

当 frame 数量非常多时，这条轨道的宽度会变成整排缩略图的总宽度；如果外层容器没有把“可滚动内容宽度”和“布局宽度”彻底隔离，主内容区就会被迫跟着使用这条超长宽度。

结果就是：

- 主图壳层仍然维持 `16:9`
- 但它的 `width: 100%` 参考的已经不是可视区域，而是那条被撑开的超长布局宽度
- 最终出现 14939px 这种离谱尺寸

### 2. 胶卷带使用了更像“拖轨道”的交互，而不是真正的原生横向滚动

旧实现依赖第三方 carousel/slider 方案去处理 filmstrip 拖动。对于 viewer 这种“底部是一条真实缩略图长带”的场景，这种方案有两个问题：

- 它更偏向“轮播轨道”而不是“可滚动素材带”
- 当缩略图数量很多时，拖拽、点击、滚动和选中之间的边界不够稳定

这也是为什么图片一多以后，“向右拖”并不能稳定到达后面的 frame。

## 当前修复方案

### 1. Swipe 主图改为受控自绘实现

当前 `Swipe` 模式不再依赖第三方 compare slider，而是改成 viewer 自己控制：

- before 图作为底层
- after 图作为覆盖层
- 用裁切区域控制 after 的显示比例
- 整个主图区域都可以拖动，不再要求用户精准抓住中间那条滑块

这样做的直接好处是：

- 主图宽高完全受 `StagePresentationShell` 约束
- 不再额外引入第三方组件的尺寸推导逻辑
- 交互响应区更大，更符合“对比图本身就是滑动操作面”的使用直觉

### 2. 胶卷带改为原生横向滚动容器

当前胶卷带不再依赖 carousel 轨道，而是改成：

- 外层 viewport：`overflow-x: auto`
- 内层 track：`width: max-content`
- 指针拖动时直接修改 `scrollLeft`

这意味着：

- 它本质上是“真正的横向滚动容器”
- 鼠标拖动、触控板滚动、滚动条滚动会汇合到同一套原生 scroll 行为
- 选中 frame 时，当前缩略图会自动 `scrollIntoView`

### 3. 布局层补强了宽度约束

除了替换交互方案，这轮还补了几个明确的约束：

- viewer 根 `Paper` 显式使用 `width: 100%` 和 `max-width: 100%`
- 主图栈和舞台容器显式加 `min-width: 0`
- 胶卷带 viewport 显式加 `width: 100%` / `min-width: 0`
- 主图壳层显式加 `max-width: 100%`

这些约束的目标很单一：**缩略图轨道可以很长，但布局宽度不能跟着一起变长。**

## 当前行为基线

修复后，viewer 应满足以下行为：

- 无论 frame 数量多少，主图都不能因为胶卷带内容过多而超出屏幕。
- 主图在普通模式下始终受当前版心约束，在 `Fit view` 模式下始终受当前浏览器视口约束。
- `Swipe` 模式下，拖动图片任意位置都应移动分割线。
- 胶卷带在 frame 很多时，必须可以持续向右拖动并滚到后面的缩略图。
- 点击缩略图仍然切换 frame；拖动滚动后不会误触发 frame 选中。

## 后续约束与经验教训

### 1. viewer 的主图不要再把“媒体比较”托付给黑盒轮播/滑块组件

这个项目的主舞台是核心交互，不适合交给第三方黑盒去自行推导尺寸和拖拽逻辑。未来即使升级更重的 viewer，也应保留：

- 主图尺寸由我们自己控制
- 主图交互边界由我们自己控制
- mode 切换时布局不跳

### 2. 胶卷带优先使用“真实滚动”，而不是“轮播模拟”

胶卷带本质上是一条素材带，不是 banner。设计原则应保持：

- 真正的横向滚动优先
- 可拖、可滚轮、可触控板
- 当前 frame 自动滚入可视区

### 3. 任何横向长内容都必须和页面版心隔离

只要组件里出现 `max-content`、长轨道、横向素材带，就要同时检查：

- 外层 viewport 是否 `overflow-x: auto`
- 版心容器是否 `min-width: 0`
- 主布局是否明确 `max-width: 100%`

否则“一个超长子元素拖垮整个布局宽度”的问题还会再来。
