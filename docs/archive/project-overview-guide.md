# Magic Compare Web 项目指南

本文档说明项目定位、边界和信息架构，作为产品与工程决策的统一基线。
遇到功能取舍时，优先以本文定义的目标与范围为准。

## 1. 项目概述

Magic Compare Web 是一个面向压制组的 **图片对比展示与内部浏览平台**。

它的目标不是替代 VSEditor，也不是在线运行 VapourSynth 脚本，更不是做一个重量级素材管理系统。

本项目主要解决以下问题：

* 将零散的对比图整理为稳定、可复用、可归档的 case
* 替代当前通过 Telegram 传图进行效果对比的低效浏览方式
* 为组内提供舒适、专业的内部浏览体验
* 为公开发布提供高质量、可长期保存的静态展示页
* 将图片顺序、说明、标签、对比模式、发布状态结构化沉淀下来

一句话概括：
**这是一个“图片对比展示与内部浏览平台”，不是视频预览器，也不是评审系统。**

---

## 2. 背景与问题

当前组内对比图交流主要通过聊天工具进行，已经暴露出明显问题：

* 图片顺序容易混乱
* 同一组对比图缺少稳定展示方式
* 手机上查看体验差，不适合认真放大观察
* 内部技术协助时，上下文容易丢失
* 没有统一的 case 页面、说明结构与归档体系
* 公开展示与内部草稿混杂，不利于长期维护

本项目旨在优先解决“展示、浏览、发布、沉淀”这四件事。

---

## 3. 项目边界

### 3.1 本项目要做的事

* 创建和展示 case、group（同一 case 下的多组对比）
* 展示多张对比图并保持稳定顺序
* 支持 before / after、overlay、heatmap 等查看模式
* 支持内部浏览、排序与发布流程
* 生成公开可访问的静态页面
* 沉淀标签、说明、元数据与案例内容

### 3.2 本项目暂时不做的事

* 在线修改 VapourSynth 脚本
* 在线运行预览渲染
* 重做 VSEditor
* 多人实时协作编辑主图
* 复杂区域级标注系统
* 完整素材管理平台
* 即时通讯

### 3.3 核心判断

如果某个功能更像“在线调图器”“在线预览器”“实时协作编辑器”，默认不进入 v1 范围。

## 4. 用户角色

### 4.1 公开访客

* 只能访问已发布的公开 case
* 只能查看，不具备任何写入权限
* 看不到草稿、内部备注和未发布内容

### 4.2 组内成员

* 可访问内部站目录
* 可通过目录浏览已导入和未公开内容

### 4.3 维护者 / 上传者

* 负责通过本地 CLI 导入图片与 metadata
* 整理顺序、标题、标签、说明与封面
* 执行发布操作

说明：
* 当前讨论、打分、评审仍然发生在 IM 聊天软件中
* 站点当前定位为“看”和“发”，不是站内讨论系统

---

## 5. 总体架构原则

### 5.1 部署原则

采用 **一个内部后台 + 一个公开发布面** 的模式。

* **内部站**：动态站点，放在 Zero Trust / Access 后面
* **公开站**：静态发布产物，只读，不提供写接口

目前后台页面只提供在 CLI 中，节约开发时间。

### 5.2 代码原则

不维护两套业务逻辑。

采用 **单仓库、共享组件与 schema、两个部署目标** 的方式：

* 共享 UI 组件
* 共享 compare viewer 核心逻辑
* 共享 case / group / image 的类型定义
* 共享主题、布局与展示型组件

### 5.3 安全原则

* 公开站不暴露上传、管理、评论、编辑等写接口
* 内部站放在 Zero Trust / Access 后面
* 默认假设公开站要承受扫描、抓取、异常请求等风险
* 默认假设内部站的重点不是抗公开攻击，而是不要裸露给外部

## 6. 技术栈决策

### 6.1 前端框架

* **Next.js App Router**
* **React**
* **TypeScript**

原因：

* 适合同时支持公开静态页和内部动态页
* 适合组件复用与分层部署
* 文档成熟，AI 生成代码命中率高
* 适合后续继续扩展交互、动效和发布流程

### 6.2 UI 与交互相关库

* **MUI**：基础 UI 壳、按钮、菜单、表单、抽屉、弹窗
* **Motion**：页面和组件级动效
* **Embla Carousel**：底部胶片式缩略图带
* **dnd-kit**：内部拖拽排序
* **react-compare-slider**：公开页轻量 before / after
* **OpenSeadragon**：内部高精度查看、缩放、overlay 能力

### 6.3 上传工具

* **Python CLI**
* 建议使用 **Typer** 组织命令

CLI 负责：

* 扫描本地目录
* 校验结构
* 生成 manifest
* 生成缩略图
* 上传图片与 metadata
* 创建或更新 case
* 管理或删除 case

Web 端先不承担复杂本地上传交互。

### 6.4 数据与存储

* 图片与缩略图：对象存储或文件系统
* case / group / image metadata：数据库存储
* 发布时生成公开静态数据

## 7. 视觉与交互设计原则

### 7.1 视觉方向

整体风格应当是：

* 深色
* 现代

使用 Frontend Design 完成进一步设计。

### 7.2 主原则

主图区域优先保证查看体验。
动效主要用于：

* 面板开合
* 缩略图切换
* 排序反馈
* 页面过渡
* 模式切换
* 工具栏与浮层反馈

不应在主图像素区域使用喧宾夺主的转场或特效。

### 7.3 页面结构方向

核心查看页采用以下布局：

* 顶部：轻量工具栏
* 中间：主图舞台
* 下方：胶片式缩略图带
* 右侧：可折叠信息侧栏

## 8. 信息架构

### 8.1 公开站

公开站没有目录页。
公开内容只通过 url 访问，且入口粒度为 **单组画廊对比页**。

#### 页面：单组对比页

完整的单组画廊对比页面。

包含：

* 主图 viewer
* 模式切换
* 对比说明
* 上下张或同组浏览能力

说明：

* 公开站不提供 case 总目录
* 不提供公开索引页
* 不暴露内部组织结构

### 8.2 内部站

#### 页面一：内部首页 / 目录页

这是内部站的主要入口，用于浏览所有 case、草稿、未公开内容和已发布内容。

#### 页面二：单组对比页

见上文 `#### 页面：单组对比页`

## 10. 核心交互功能

### 10.1 查看模式

所有模式都要：
* 支持触控
* 支持键盘

系统至少支持以下模式：

#### Before / After Slider

* 使用切割线滑动

#### Overlay / Heatmap

* heatmap 叠加
* 支持开关
* 支持透明度调节

#### A / B 切换

**最重要的看图模式**

* 快捷切换原图与处理图
* 用于整体和细节观察

Zoom Compare
* 适用于内部高精度查看
* 支持缩放和平移

### 10.2 缩略图带

下方缩略图带是核心导航组件。

要求：
* 横向滚动顺畅
* 当前选中态明显
* 支持键盘切换
* 内部模式下支持拖拽排序
* 选中和切换动效要顺滑但克制

### 10.3 右侧侧栏

右侧信息栏可折叠，按模式显示：
* group 列表
* 当前图片说明
* 元数据
* 发布状态

## 11. 上传与导入流程

### 11.1 上传原则

图片一定发生在本地，因此通过本地 Python CLI 导入。

### 11.2 CLI 职责

CLI 应完成：

* 扫描目录
* 读取约定结构
* 校验文件命名和组织
* 生成 manifest
* 自动生成缩略图 (todo: 使用 webp 优化缩略图)
* 调用内部后台接口完成导入或同步

---

## 12. 发布流程

### 12.1 原则

发布必须是显式操作。
内部/公开内容必须在状态上严格区分。

### 12.2 发布动作应完成的事情

1. 校验 case 是否完整
2. 过滤出允许公开的内容
3. 生成公开站所需 metadata
4. 同步公开图片与缩略图
5. 生成或更新公开静态页面数据
6. 更新公开站索引

### 12.3 公开站原则

公开站只消费“发布产物”，不直接访问内部后台写接口。

## 14. AI 开发要求

### 14.1 总体要求

* 先搭建稳定架构，再逐步叠功能
* 先定数据模型和页面骨架，再做视觉打磨
* 禁止写成一次性原型
* 组件职责必须明确

### 14.2 编码要求

* 使用 TypeScript 严格模式
* 目录结构清晰
* 命名统一
* 重要逻辑和状态流有清晰注释
* 禁止把大量逻辑塞进单一页面文件
* 优先低耦合、可复用设计

### 14.3 UI 要求

* 默认以深色 viewer 工作台为目标
* 不要套用通用 admin 模板
* 动效应精致、轻巧、克制
* 主图区域优先保证判断精度
