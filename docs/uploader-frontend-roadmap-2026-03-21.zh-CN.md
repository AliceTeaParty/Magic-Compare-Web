# Uploader Tool 与前端网页能力评估与路线图（2026-03-21）

本文面向当前仓库已实现能力，给出：

- 现状评估（uploader / 前端）
- 必需增加项（建议优先做）
- 可选增强项（体验和效率）
- 分阶段路线图（含验收标准）

## 1. 当前现状（基于仓库实现）

## 1.1 uploader-tool

当前 `tools/uploader` 已具备：

- 本地目录扫描、命名解析、结构校验
- 自动生成缩略图与 heatmap
- 上传素材到 S3-compatible 存储
- 生成并提交 `ImportManifest` 到 internal-site
- 中文向导式入口（`magic-compare-uploader`）
- `.env` 配置与 Cloudflare Access 认证流程

结论：**核心链路可用且闭环**，已能从“本地素材”走到“内部站可查看”。

## 1.2 前端网页（internal/public）

当前前端已具备：

- internal-site：case/workspace/viewer、导入同步、发布、公开导出与部署入口
- public-site：静态只读消费 `content/published` 的公开页面
- viewer 三模式（Slider / A/B / Heatmap）与基础工作区操作

结论：**主流程完整**，但在“可观测性、批量效率、稳定性反馈、协作体验”上还有明显提升空间。

## 2. 必需增加项（P0 / P1）

以下为“建议必须补齐”的能力，优先保证可运营与可排障。

## 2.1 uploader-tool（必需）

1. **Dry-run / Plan 模式（P0）**
   - 目标：只做扫描、校验、生成执行计划，不上传不写远端。
   - 价值：导入前可预估风险，降低误操作成本。
   - 验收：输出待上传文件数、缺失项、命名冲突与预计目标路径，退出码可用于 CI。

2. **结构化错误报告（P0）**
   - 目标：除 CLI 彩色输出外，额外输出机器可读报告（JSON）。
   - 价值：便于自动化收集失败原因、做批量修复工具。
   - 验收：`--report-json <path>` 生成报告，包含 case/group/frame 级错误。

3. **断点续传与跳过已存在对象（P1）**
   - 目标：重复执行时可复用已上传对象，减少网络与时间成本。
   - 价值：大批量导入时显著提升稳定性与效率。
   - 验收：可选 `--resume` / `--skip-existing`，重复执行时耗时明显下降。

4. **导入后校验（P1）**
   - 目标：sync 后自动抽样校验关键 URL 可访问、manifest 可被 internal-site 正常消费。
   - 价值：把“上传成功但页面打不开”的问题前置到 CLI 阶段。
   - 验收：校验失败时给出可复现定位信息（路径、URL、HTTP 状态）。

## 2.2 前端网页（必需）

1. **导入任务可观测性（P0）**
   - 目标：在 internal-site 提供“最近导入记录 / 状态 / 错误摘要”。
   - 价值：减少“命令行黑盒”感，便于运营和排障。
   - 验收：至少可看到最近 N 次导入的时间、来源、成功/失败、失败原因摘要。

2. **资源健康检查视图（P1）**
   - 目标：在 workspace 标记素材缺失、热力图缺失、URL 异常。
   - 价值：发布前发现内容缺陷，降低线上坏链。
   - 验收：提供可筛选异常项，并支持跳转到对应 group/frame。

3. **发布前检查清单（P1）**
   - 目标：publish 前给出规则检查（是否公开、封面是否存在、关键素材是否齐全）。
   - 价值：减少返工与误发布。
   - 验收：未通过时阻断发布并给出明确修复建议。

## 3. 可选增强项（P2，按收益排期）

## 3.1 uploader-tool（可选）

- **并发与速率控制**：`--concurrency`、失败重试策略可配置。
- **批量模板生成器**：从平铺目录一键生成规范目录骨架与 metadata 草稿。
- **跨平台打包发布**：提供 zip/installer，降低 Python 环境门槛。
- **插件化预处理钩子**：允许接入自定义重命名/元数据补全逻辑。

## 3.2 前端网页（可选）

- **导入向导页（轻量版）**：网页内展示 uploader 命令建议与参数预填，不做浏览器大文件上传。
- **批量操作增强**：group/frame 批量公开开关、批量排序调整。
- **Viewer 专业功能**：关键帧标记、差异阈值切换、快捷键帮助浮层。
- **性能体验增强**：大 case 下虚拟化列表、懒加载策略可视化、首屏加载指标面板。

## 4. 建议路线图（8~10 周）

## 阶段 A（第 1-2 周）稳定性与可观测性基础

- uploader：Dry-run、结构化报告
- 前端：导入记录最小可视化（列表 + 详情抽屉）

阶段验收：

- 导入失败可从“命令行错误”定位到“结构化原因”
- internal-site 可查看近期导入结果

## 阶段 B（第 3-5 周）效率与质量门禁

- uploader：`--resume` / `--skip-existing`、导入后校验
- 前端：资源健康检查、发布前检查清单

阶段验收：

- 重复导入时长明显下降
- 发布前可拦截缺图/坏链等高频问题

## 阶段 C（第 6-8 周）体验提升（可选项打包）

- uploader：并发调优、模板生成器
- 前端：批量操作、viewer 体验增强

阶段验收：

- 批量 case 操作步骤减少
- 大数据量场景下交互更稳定

## 5. 里程碑指标（建议）

- 导入失败平均排障时间（MTTR）下降 ≥ 40%
- 重复导入平均耗时下降 ≥ 30%
- 发布后因资源缺失导致的回滚次数下降 ≥ 60%
- workspace 首屏可交互时间（大 case）下降 ≥ 20%

## 6. 实施原则

- 保持“本地优先导入 + 站内可观测”边界，不把 internal-site 变成重型在线上传器。
- uploader 与 internal-site 的契约优先通过 schema 与结构化报告稳定化。
- 优先交付能减少排障成本的能力，再投入高阶体验增强。
