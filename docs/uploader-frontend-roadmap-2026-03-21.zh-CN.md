# Uploader Tool 与前端网页能力评估与路线图（2026-03-21）

本文基于当前仓库实现，重点评估 uploader-tool 还应优先补齐哪些能力，以及前端 / 服务端只需要提供哪些最小必要配套。
核心原则是先把“导入更稳、更好用”做好，再补站内的轻量可观测与发布门禁。

## 1. 当前现状（基于仓库实现）

## 1.1 uploader-tool

当前 `tools/uploader` 已具备：

- 本地目录扫描、命名解析、结构校验
- 自动生成缩略图与 heatmap
- 上传素材到 S3-compatible 存储
- 生成并提交 `ImportManifest` 到 internal-site
- 中文向导式入口（`magic-compare-uploader`）
- `.env` 配置与 Cloudflare Access 认证流程

结论：**核心链路已经闭环**，但“上传前发现问题”“上传过程体验”“失败后复盘成本”仍然偏高。

## 1.2 前端网页与站内配套

当前前端已具备：

- internal-site：case/workspace/viewer、导入同步、发布、公开导出与部署入口
- public-site：静态只读消费 published bundle 的公开页面
- viewer 基础三模式与工作区主流程

结论：**主流程够用**。短期不建议继续展开大量前端功能细节，重点是给 uploader 提供最小必要的站内配套，而不是把 internal-site 继续做重。

## 2. 必需增加项（P0 / P1）

## 2.1 uploader-tool（重点）

1. **Dry-run / Plan 模式（P0）**
   - 目标：只做扫描、校验、生成执行计划，不上传不写远端。
   - 价值：在真正上传前就能发现命名、结构、缺失项问题。
   - 验收：输出待上传文件数、缺失项、命名冲突与目标路径摘要，退出码可用于自动化。

2. **结构化错误报告（P0）**
   - 目标：除 CLI 彩色输出外，额外输出机器可读报告（JSON）。
   - 价值：便于失败后复盘、批量修复和自动化收集。
   - 验收：`--report-json <path>` 可输出 case / group / frame 级问题。

3. **上传客户端体验增强（P1）**
   - 目标：让长时间上传任务更可控，而不是“等命令跑完再看结果”。
   - 建议方向：阶段进度、结果摘要、失败重试、跳过已存在对象、断点续传。
   - 验收：重复执行与大批量导入时，失败恢复成本明显下降。

4. **图片解码校验（P1）**
   - 目标：把“文件存在但其实坏掉 / 不是正常图片”的问题尽量前置。
   - 本地侧：在 uploader 上传前，对关键图片做快速解码校验。
   - 服务端侧：在 import / publish 链路做轻量 sanity check，只确认内容不是明显损坏或伪装文件。
   - 边界：这是小项目场景下的简单校验，不需要设计成复杂安全系统。

## 2.2 前端 / 服务端最小配套

1. **导入结果可见性（P1）**
   - 目标：在 internal-site 至少能看到最近导入结果与失败摘要。
   - 价值：减少 uploader 失败后只能回终端翻日志的情况。

2. **发布前的轻量检查（P1）**
   - 目标：在发布前提示明显问题，如关键素材缺失、公开内容不完整。
   - 价值：把高频返工挡在发布前，而不是发布后再回头修。

说明：

- 这里的站内能力以“最小必要” 为主，不建议把 internal-site 扩成重型在线上传后台。
- 服务端配套应尽量压缩，只承担校验、记录和发布门禁，不承担复杂上传编排。

## 3. 可选增强项（P2）

## 3.1 uploader-tool

- **并发与速率控制**：允许针对不同网络环境调整上传节奏。
- **批量模板生成器**：从平铺目录快速生成规范骨架与 metadata 草稿。
- **更友好的分发方式**：提供 zip / installer，降低 Python 环境门槛。
- **可插拔预处理钩子**：便于接入自定义重命名或 metadata 补全过程。

## 3.2 前端网页

- **轻量命令辅助页**：在网页内提示 uploader 命令和参数建议，不做浏览器大文件上传。
- **批量操作增强**：针对 group / frame 做更省步骤的批量处理。
- **Viewer 体验设想**：继续打磨交互和专业观察体验，但不在本阶段展开实现细节。

## 4. 建议路线图（8~10 周）

## 阶段 A（第 1-2 周）先把上传链路做稳

- uploader：Dry-run、结构化错误报告、图片本地解码校验
- 配套：服务端轻量 sanity check

阶段验收：

- 上传前即可发现大部分结构和图片损坏问题
- 上传失败后能快速定位到具体原因

## 阶段 B（第 3-5 周）提升上传体验与站内配套

- uploader：上传进度与摘要、失败恢复、跳过已存在对象 / 断点续传
- internal-site：最近导入结果可见、发布前轻量检查

阶段验收：

- 重复导入和失败重跑的成本明显下降
- 发布前能拦截高频缺图 / 坏链问题

## 阶段 C（第 6-8 周）按收益补体验

- uploader：模板生成器、更友好的分发方式
- 前端：批量操作与 viewer 体验增强

阶段验收：

- 新人上手 uploader 成本降低
- 大批量 case 的整理和发布步骤减少

## 5. 里程碑指标（建议）

- 导入失败平均排障时间（MTTR）下降 ≥ 40%
- 重复导入平均耗时下降 ≥ 30%
- 因图片损坏或资源异常导致的返工次数明显下降
- 从本地素材到 internal-site 可查看的稳定性明显提升

## 6. 实施原则

- uploader-tool 是近阶段的主投入方向，前端 / 服务端只补最小必要能力。
- 先解决“上传前发现问题、上传中可控、上传后可复盘”，再考虑高级体验。
- 本地校验优先，服务端做简单兜底，不把小项目做成复杂安全平台。
- internal-site 继续保持“内部工作站 + 控制面”定位，不扩成重型上传系统。
